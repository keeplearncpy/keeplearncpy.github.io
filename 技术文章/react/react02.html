# 一、 React脚手架(Scaffold)

  • 脚手架是一个自动化的项目构建工具，通过脚手架可以快速构建一个项目的基础结构，从而可以快速开发项目。

# 1. 使用脚手架创建项目

create-react-app

  • 不管是vue脚手架还是angluar或者react的脚手架，这些脚手架都是基于Node环境，并且也是基于webpack。
  • 使用脚手架创建项目
// 全局安装脚手架
npm install create-react-app -g
// 创建项目
create-react-app 项目名称
//注意项目名称不能有大写字母（驼峰命名）

# 2. PWA

  • 在使用react脚手架创建项目时，会支持PWA，在vue使用脚手架创建项目时，也会有支持PWA这个选项。

# 2.1 什么是PWA

  • PWA全称是Progressive Web App，即渐进式WEB应用。
  • 一个PWA应用首先是一个网页，可以通过Web技术编写出一个网页应用。
  • 随后添加上App Manifest和Service Worker来实现PWA的安装和离线等功能，这种既是Web app。

# 2.2 PWA解决了哪些问题

  • 可以添加到主屏幕上
  • 是实现离线缓存功能，即使用户手机没有网络，依然可以使用一些离线功能
  • 实现了消息推送
  • 一系列类似于Natvie app 相关的功能

# 二、 React组件化开发

# 1. 什么是组件化开发（分而治之）

  • 组件化开发是一种软件开发思想，它将一个复杂的系统分解成多个模块化的功能单元，每个单元都是一个组件，组件之间可以相互组合，从而快速构建出复杂的应用，并且整个页面的管理和维护变得更加容易。
  • 组件化开发的好处
    • 提高开发效率
    • 提高代码的复用性
    • 降低维护成本

# 2. React组件化（核心思想之一）

  • React组件相对于Vue更加的灵活，根据不同的方式可以分成很多的类组件：
    • 根据组件的定义方式，可以分为类组件（Class Component）和函数组件（Functional Component） -根据组件内部是否有状态需要维护，可以分为：有状态组件（Stateful Component）和无状态组件（Stateless Component）
    • 根据组件的使用方式，可以分为：容器组件（Container Component）和展示组件（Presentational Component）
    • 根据组件的功能，可以分为：高阶组件（Higher-Order Component）和受控组件（Controlled Component） -......
  • 这些组件虽然很多，但实际他们最主要就是关注数据逻辑和UI展示的分离：
    • 函数组件、无状态组件、展示组件主要关注UI的展示
    • 类组件、有状态组件、容器组件主要关注数据的逻辑

# 3. 类组件

  • 类组件是React中最基本的组件，它是一个类，这个类继承自React.Component，这个类中必须要有一个render方法，这个render方法必须要有一个返回值，这个返回值就是这个组件要渲染的内容。
  • 类组件的定义方式
Class App extends React.Component{
    constructor(){
        super()
        this.state = {} // 初始化状态,维护数据状态
    }
    // 必须要有一个render方法，这个render方法必须要有一个返回值，这个返回值就是这个组件要渲染的内容
    render(){
        return (
            <div>类组件</div>
        )
    }
}

  1. React元素：通过JSX创建，例如
    会被渲染成DOM节点,会被渲染成自定义组件
  2. 数组或者fragments：使得render函数可以返回多个元素
  3. Portals：使得子节点可以渲染到父组件之外的DOM节点上
  4. 字符串或者数值类型：它们在DOM中会被渲染成文本节点
  5. 布尔类型或者null：什么都不渲染

# 5. 函数组件

  • 函数组件是一种定义组件的方式，它是一个函数，这个函数的返回值就是这个组件要渲染的内容。
  • 函数组件的定义方式
function App(){
    return (
        <div>函数组件</div>
    )

  • 函数组件和类组件的区别（没有hooks）
    • 函数组件没有this，没有生命周期，没有状态，没有上下文，没有实例方法
    • 类组件有this，有生命周期，有状态，有上下文，有实例方法
    • 函数组件的性能比类组件的性能要高
    • 函数组件没有this.state，没有this.setState，没有this.props，没有this.context，不能管理状态

# 三、 生命周期（类组件的生命周期，函数组件没有）

# 1. 什么是生命周期

  • 生命周期是组件从创建到销毁的整个过程，React组件的生命周期分为三个阶段：
    • 组件的挂载阶段 --- componentDidMount函数：组件挂载到DOM上时调用
    • 组件的更新阶段 --- componentDidUpdate函数：组件更新时调用
    • 组件的卸载阶段 --- componentWillUnmount函数：组件从DOM上卸载时调用

# 2. 常见的生命周期

  • componentDidMount：组件挂载到DOM上时调用
  • 在这个函数中通常进行哪些操作？
    • 发送网络请求（官方建议）
    • 添加订阅（会在componentWillUnmount取消订阅）
    • 设置定时器
    • DOM操作
  • componentDidUpdate：组件更新时调用（首次渲染不会执行此方法）
    • 当组件更新后，可以在此处对DOM进行操作；
    • 但是要注意，如果在此处进行setState操作，会导致组件更新，从而会导致死循环。
    • 如果你对更新前后的props进行了比较，那么也可以在此处进行网络请求。（props没有发生变化时，则不会发送网络请求）
  • componentWillUnmount：组件从DOM上卸载时调用
    • 在这个函数中通常进行哪些操作？
      • 取消网络请求
      • 取消订阅
      • 清除定时器
      • DOM操作

# 3. 不常见的生命周期

  • getSnapshotBeforeUpdate：在组件更新前调用，可以获取DOM更新前的一些信息（比如说滚动位置），返回值将作为componentDidUpdate的第三个参数。
  • shouldComponentUpdate：（很常用）组件更新前调用，返回值决定是否更新组件，如果返回false，则不会更新组件，这个函数主要用于性能优化。
  • getDerivedStateFromProps：state的值在任何时候都依赖于props，该方法返回一个对象来更新state，如果返回null，则不更新任何内容。

// 父组件
import React, { Component } from 'react'
import AddCounter from './AddCounter'
import SubCounter from './SubCounter'

export default class App extends Component {
    constructor(){
        super()
        this.state = {
            counter: 0
        }
    }
    changeCount(count){
        this.setState({
            counter: this.state.counter + count
        })
    }
    subCount(count){
        this.setState({
            counter: this.state.counter - count
        })
    }
  render() {
    const {counter} = this.state
    return (
      <div>
        <div>子传父</div>
        <div>当前计数：{counter}</div>
        <AddCounter addClick={(count) => this.changeCount(count)}/>
        <SubCounter subClickCount={(count) => this.subCount(count)}/>
      </div>
    )
  }
}

// 子组件

}

# 五、 类似VUE插槽效果

# 1. 什么是插槽

  • 插槽是一种组件的设计模式，它可以让一个组件的内容可以由其父组件决定。
  • React对于插槽的支持非常灵活，有两种方案可以实现
    • 使用children属性
    • 使用 props 属性传递React元素

# 2. 使用children属性

  • children属性是一个特殊的属性，它可以让父组件可以将子组件的内容传递给子组件。
  • 代码演示
// 父组件
import React, { Component } from 'react'
import Tabto from './Tabto'
export class App extends Component {
  render() {
    return (
      <div>
        {/**方式1：通过children属性实现插槽 */}
        <Tabto >
          <button>按钮·</button>
          <a>链接</a>
          <span>span</span>
        </Tabto>
      {/**方式2：使用props */}
      <Tabto left={<button>按钮</button>} center={<a>链接</a>} right={<span>span</span>}></Tabto>
      </div>
    )
  }
}

export default App

# 六、 Context应用场景

# 1.非父子组件数据的共享

  • 在开发中，比较常见的数据传递方式是通过props属性自上而下（由父到子）进行传递。

  • 但是对于写场景：比如一些数据需要在多个组件中进行共享（地区偏好、UI主题、用户登录状态、用户信息等）。

  • 如果我们在顶层的APP中定义这些信息，然后一层层传递下去，那么对于一些中间层不需要这些数据的组件来说，是一种冗余操作。

  • 使用Context---这个API（后面会有redux来专门维护这些数据）

  • Context提供了一种在组件之间共享此类值的方式，而不必显示地通过组件书的逐层传递props

  • Context设计的目的就是为例共享那些对于一个组件树而言是“全局”的数据，例如用户的token

# 2. Context相关API

  • React.createContext：创建一个Context对象，当React渲染一个订阅了这个Context对象的组件，这个组件会从组件树中离自身最近的那个匹配的Provider中读取到当前的context值。

  • Context.Provider：每个Context对象都会返回一个Provider React组件，它允许消费组件订阅context的变化。

    • Provider接收一个value属性，传递给消费组件。一个Provider可以和多个消费组件有对应关系。多个Provider也可以嵌套使用，里层的会覆盖外层的数据。
    • 当Provider的value值发生变化时，它内部的所有消费组件都会重新渲染。Provider及其内部consumer组件都不受制于shouldComponentUpdate函数，因此当consumer组件在其祖先组件退出更新的情况下也能更新。

  • Class.contextType：挂载在class上的contextType属性会被重赋值为一个由React.createContext()创建的Context对象。这能让你使用this.context来消费最近Context上的那个值。你可以在任何生命周期中访问到它，包括render函数中。

  • Context.Consumer：这里，React组件也可以订阅到context变更。这能让你在函数式组件中完成订阅context。

    • 这里需要函数作为子元素（function as a child）这种做法。
    • 这个函数接收当前的context值，返回一个React节点。
    • 传递给函数的value值等同于往上组件树离这个context最近的Provider提供的value值。如果没有对应的Provider，value参数等同于传递给createContext()的defaultValue。

  • Context.displayName：context对象接受一个名为displayName的property，类型为字符串。React DevTools使用该字符串来确定context要显示的内容。

  • 代码演示

// 1.创建Context对象（也可以专门在js文件里面去创建这种共享对象）

// 1.创建Context对象createContext里面的值是默认的defaultValue

    nickname: "默认",
    level: -1,

# 3. Context的使用注意事项

  • Context主要应用于一些跨层级的组件间通信，如果你只是想避免层层传递一些属性，组件组合（component composition）有时候是一个比context更好的解决方案。
  • 不要在生产环境中使用context，这会导致你的应用程序很难升级。
  • Context.Consumer和Class.contextType相比，后者只能订阅单一的context来源，而前者可以订阅多个context。
  • 如果你在使用contextType，那么建议使用静态属性contextType而不是Context.Consumer。
  • 如果你使用了contextType，那么在编写context时，建议提供一个默认值。这样可以在没有对应的Provider的情况下，也能消费到某些值。
  • 什么时候使用Context.Consumer
    • 当你需要在一个函数式组件中订阅context的变化时，或者你不喜欢渲染一个函数作为一个子组件，那么你可以使用Context.Consumer。
    • 当组件中需要使用多个Context时，使用Context.Consumer。
